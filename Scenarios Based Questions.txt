🚀 Scenario 1: Kafka Broker Crash + In-Flight Messages
❓ Situation

A Kafka broker crashes while producers are sending policy_created events.

✅ What really happens?
Kafka stores data in replicated partitions
One broker is leader, others are followers

👉 If leader crashes:

A follower becomes new leader automatically
No data loss if replication is configured correctly
✅ Strong Answer

In our system, we configure Kafka with replication and set producer acknowledgment to acks=all. This ensures that a message is only considered successful when it is written to all replicas. If a broker crashes, another replica becomes leader and continues serving requests without data loss.

⚠️ Failure Risk
If acks=1 → possible data loss
If replication factor = 1 → high risk
🚀 Scenario 2: Consumer Crashes After Processing but Before Commit
❓ Situation

Payment Service processes payment but crashes before committing offset.

🔥 What happens?
Kafka thinks message is not processed
Message will be reprocessed

👉 Result: Duplicate processing

✅ Strong Answer

This leads to duplicate message processing because Kafka uses at-least-once delivery. To handle this, we implement idempotency using unique transaction IDs and database checks to ensure the same payment is not processed twice.

✔ Solution
Idempotent consumer
Unique transaction ID
DB constraint

🚀 Scenario 4: SQS Message Not Deleted (Visibility Timeout Issue)
❓ Situation

Lambda processes message but fails to delete it.

🔥 What happens?
Message becomes visible again
Another consumer processes it

👉 Duplicate execution

✅ Strong Answer

In SQS, if a message is not deleted within the visibility timeout, it becomes visible again and can be reprocessed. To handle this, we design idempotent consumers and use proper visibility timeout settings to ensure the processing completes before the message becomes visible again.

✔ Best Practices
Increase visibility timeout
Use idempotency
🚀 Scenario 5: Lambda Fails During Processing
❓ Situation

Lambda triggered by SQS fails midway.

🔥 What happens?
Message goes back to queue
Retry happens
✅ Strong Answer

Lambda automatically retries processing SQS messages if it fails. After a configured number of retries, the message is sent to a Dead Letter Queue (DLQ) for further analysis. This ensures no message is lost and failed messages can be handled separately.

✔ Design
SQS → Lambda → Fail → Retry → DLQ
🚀 Scenario 6: Ordering Problem in Kafka
❓ Situation

Multiple partitions

Kafka guarantees ordering only within a partition. To maintain order for related events like a specific policy, we use a key such as policyId so that all related messages go to the same partition and maintain order.

🚀 Scenario 8: Duplicate Messages Across Kafka + SQS
❓ Situation

Same message processed multiple times across systems.

✅ Strong Answer

Since both Kafka and SQS provide at-least-once delivery, duplicates are possible. We handle this using idempotent design, where each message has a unique ID and we check against the database before processing to ensure it is not executed multiple times.

🚀 Scenario 9: Partial Failure in Saga
❓ Situation

Policy created but payment fails.

✅ Strong Answer

We use Saga pattern with compensating transactions. If payment fails after policy creation, we trigger a compensation action to cancel or mark the policy as failed, ensuring data consistency across services.

🚀 Scenario 10: Network Partition (Split Brain)
❓ Situation

🔥 Risk

✅ Strong Answer

Kafka uses leader election and in-sync replicas to avoid split-brain scenarios. Only one leader is active for a partition, and writes are accepted only by the leader, ensuring consistency.
